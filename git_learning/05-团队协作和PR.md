# 05 团队协作和 PR

这一节学习 GitHub 上常见的合作方式。

## 什么是 Pull Request

Pull Request 通常简称 PR。

它的意思是：我在一个分支上完成了修改，现在请求把这些修改合并到目标分支。

比如：

```text
docs/git-learning -> main
```

意思是：请求把 `docs/git-learning` 分支合并到 `main` 分支。

## 为什么不直接改 main

在团队项目里，`main` 通常代表稳定代码。

如果所有人都直接往 `main` 上提交，容易出现：

- 代码没检查就上线
- 多个人修改互相影响
- 出问题后不知道是谁引入的
- 很难做代码审查

所以更常见的流程是：

1. 从 `main` 创建新分支
2. 在新分支完成修改
3. 推送新分支到 GitHub
4. 创建 Pull Request
5. 等别人 Review
6. 通过后合并到 `main`

## 协作开发常见命令

从主分支拉取最新代码：

```bash
git switch main
git pull
```

创建自己的任务分支：

```bash
git switch -c feature/your-task
```

提交修改：

```bash
git add .
git commit -m "完成某个功能"
```

推送分支：

```bash
git push -u origin feature/your-task
```

然后去 GitHub 页面创建 Pull Request。

## 什么是 Code Review

Code Review 就是代码审查。

别人会看你的 PR，关注：

- 代码能不能运行
- 有没有明显 bug
- 命名是否清楚
- 是否影响其他功能
- 是否需要补充说明或测试

对初学者来说，Review 不是挑刺，而是帮你把代码变得更可靠。

## 什么是冲突

冲突通常发生在两个人修改了同一个文件的同一部分。

Git 不知道应该保留谁的修改，就会提示冲突。

冲突不是 Git 坏了，也不是你做错了，而是需要人工判断最终内容。

冲突文件里可能会出现：

```text
<<<<<<< HEAD
当前分支的内容
=======
另一个分支的内容
>>>>>>> branch-name
```

你需要手动编辑成最终想保留的样子，然后：

```bash
git add 冲突文件
git commit
```

## 初学者要记住

PR 的核心价值不是“多一个步骤”，而是让代码在进入主分支前有一次检查机会。

面试时如果你能说清楚分支开发、PR、Review、冲突解决，会比只说“我会 git add 和 git commit”更有说服力。
